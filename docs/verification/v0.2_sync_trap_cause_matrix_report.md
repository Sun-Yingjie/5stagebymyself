# v0.2 同步异常 Cause 矩阵验证报告

## 1. 结论

- 验证日期：2026-07-18；
- 基线提交：`151ff0c`（PR #4 合并后的 `main`）；
- 测试实现提交：`bb0ba43`；
- 本地工具：Icarus Verilog 13.0、Verilator 5.048；
- 处理器 RTL 改动：无；
- 叶子模块回归：14/14 testbench 通过；
- core 定向回归：20/20 场景通过；
- 架构事件：139 条普通退休、12 次同步 trap、21 笔 DMem 请求；
- 自动检查：3918 项；
- 可综合顶层 Verilator `-Wall` lint：通过。

`docs/02_pipeline_contract.md` 定义的同步异常 cause 0、1、2、3、4、5、6、7、11
现在都至少有一个 core 级 self-checking 场景。叶子测试继续负责穷举检测条件，core
测试负责证明异常从产生、流水携带、MEM 提交到重定向和副作用抑制的完整路径。

本结果关闭 `docs/04_verification_and_asic_plan.md` 中 G3A 的 directed cause-matrix
门禁，但不代表 v0.2 已冻结。ACT4、可变延迟与广泛 backpressure 门禁以及正式发布
记录仍未完成。

## 2. 本增量范围

本增量只增加验证能力：

- IMem 模型可在合法地址返回指定指令字并同时报告 error；
- core scoreboard 精确统计 trap-vector 请求，证明 handler 首条指令只请求和退休一次；
- 新增 cause 0、1、3、4、6、7 的 core 场景；
- decoder 增加 `EBREAK` 非零 `rd/rs1` 的非法编码负例；
- 所有 trap 场景继续检查精确 payload、普通退休、DMem 请求和年轻副作用。

本增量不实现 `MRET`、Machine counter、interrupt、RV32M，不扩展 CSR profile，
不接入 ACT4、差分或新的随机 backpressure 框架，也不修改 NPU、互联和 AMBA 规划。

## 3. 分层验证策略

同步异常使用统一通路：

```text
IF / ID / EX 检测
        ↓ exception metadata
MEM 最终异常合并
        ↓
trap 提交 + CSR 更新 + mtvec redirect
        ↓
异常指令和年轻指令无普通副作用
```

叶子 TB 检查“异常是否在正确位置产生”，core TB 检查“异常进入真实流水后是否精确
提交”。例如 taken branch、JAL、JALR 的错位目标和 not-taken branch 已在 EXU
逐项验证；core 再选择 JALR 作为 cause 0 的代表，验证完整 trap 提交通路。这样既不
重复复制同一流水测试，又不把不同异常源只当成代码推断。

## 4. Cause 矩阵

| Cause | 异常 | Core 证据 | 关键正向与负向检查 |
|---:|---|---|---|
| 0 | instruction address misaligned | `control_address_misaligned` | not-taken `BEQ +2` 正常退休；JALR 先清 bit 0 后以目标 2 trap；不写 `rd`、不产生普通 redirect、年轻 store 消失 |
| 1 | instruction access fault | `instruction_access_fault` | PC 4 返回合法 store 指令字和 IMem error；cause 1 原样携带到 MEM；poisoned store 的访存控制被清除且不发 DMem 请求 |
| 2 | illegal instruction | `precise_illegal_trap`、两个 CSR illegal 场景 | `trap_value` 为完整指令字；老 WB 可退休；异常与年轻 store 不退休、不产生副作用 |
| 3 | breakpoint | `breakpoint_trap` | `EBREAK` 精确产生 cause 3、value 0；不误报 illegal；年轻 store 消失 |
| 4 | load address misaligned | `load_address_misaligned` | `LW` 地址 2 产生 cause 4、value 2；不发 DMem 请求、不写 `rd` |
| 5 | load access fault | `dmem_fault_trap_wait` | 等错误响应真实完成后才 trap；无错误写回、事务最终排空 |
| 6 | store/AMO address misaligned | `store_address_misaligned` | `SW` 地址 2 产生 cause 6、value 2；不发 DMem 请求、不修改内存 |
| 7 | store/AMO access fault | `store_access_fault` | 越界 store 只发一笔请求；响应等待期间不提前 trap；错误响应同拍阻止年轻合法 store；内存不变、事务排空 |
| 11 | environment call from M-mode | `trap_redirect_backpressure` | `ECALL` 产生 cause 11、value 0；trap target 请求在 backpressure 下保持稳定并只握手一次 |

完整 SYSTEM 指令匹配还有 decoder 负例：`ECALL/EBREAK` 的非零 `rd/rs1`、保留
`funct3` 和当前未实现的 `MRET` 都不能被误识别为环境调用或断点。

## 5. 精确性证据

### 5.1 正向事件

每个 core trap 场景都登记期望的 `trap_pc/cause/value`。本轮新增的六个 cause 场景
还在 trap 提交后直接检查内部 `mepc/mcause/mtval` 与同一 payload 一致；任何 cause、
PC 或附加值错误都会使场景失败。早期异常相对后级重新检测的优先级继续由 IDU、EXU
和 LSU 叶子测试逐级检查。

### 5.2 负向事件

scoreboard 对普通退休、DMem 请求和 trap 都采用精确计数与顺序比较：

- 异常指令若普通退休，会成为未期望退休并立即报错；
- 未对齐或 poisoned 访存若发出请求，会成为未期望 DMem 请求；
- 年轻 store 的目标内存必须保持原值；
- trap 后 LSU 不得留下 phantom outstanding transaction；
- trap-vector 请求数与 handler 首条指令退休数都必须恰好为一。

### 5.3 Store access fault 竞争

cause 7 场景先暂停 DMem response，使越界 store 位于 MEM、年轻合法 store 位于 EX。
错误响应释放时，存储模型同拍重新具备接收下一请求的能力。测试确认最终 MEM 异常通过
`ex_request_block` 在 LSU 内部阻止年轻请求，只有越界 store 的第一笔请求发生。

DMem 模型对越界请求返回 error 且不执行写入。这验证了当前存储适配器合同；未来连接
真实总线时仍必须保证“报告失败的 store 没有形成不可撤销的架构可见写入”，因为 core
不能回滚外部已经完成的写副作用。

## 6. 复现命令

完整 Icarus 叶子回归和 Icarus/Verilator core 回归：

```bash
scripts/run_v0_1_regression.sh
```

可综合 core 顶层 lint：

```bash
verilator --lint-only --sv -Wall -Wno-fatal \
  -Wno-DECLFILENAME -Wno-UNUSEDSIGNAL -Wno-UNUSEDPARAM \
  --top-module rv32_core -f filelists/rv32_core_rtl.f
```

预期 core 摘要：

```text
[PASS] rv32_core: 20/20 scenarios, 139 retirements, 12 traps, 21 DMem requests, 3918 checks
```

## 7. 尚未关闭的 v0.2 门禁

当前不能创建 `v0.2-rv32i-trap` tag、Release 或冻结记录，原因包括：

- 尚未用锁定版本和配置的 ACT4 运行适用 RV32I 与异常路径测试；
- 当前只有有限定向 backpressure，尚未形成任意等待和广泛随机 backpressure 证据；
- 尚未在干净、CI 通过的 `main` 上执行冻结记录、annotated tag 和 Release 流程。

因此下一条主线应为：

```text
同步异常 directed cause matrix（本报告）
    → test/act4-rv32i
    → 补齐或明确可变延迟与 backpressure 门禁
    → v0.2 freeze record / tag / Release
    → MRET 与 Machine counter
```

ACT4 runner 首次接入只服务 v0.2 的 RV32I 与适用异常计划；后续再复用同一 runner
扩展 v0.3 Zicsr/M-mode、v0.4 RV32M 和 v0.5 interrupt/privileged 测试。
