# 处理器核系统上下文与外部接口

> 文档状态：v0.1 草案  
> 当前阶段：先定义处理器核与外部世界的边界，暂不确定每一根 RTL 信号。

## 1. 处理器核的系统定位

`rv32_core` 是一个可复用的处理器核 IP，而不是完整 SoC。核心负责执行指令和维护处理器架构状态；存储器、总线适配器、外设、NPU 数据通路和芯片级复位电路位于核心之外。

```text
                         时钟、核心同步复位
                                  │
                                  ▼
┌────────────────┐      ┌────────────────────┐      ┌────────────────┐
│ 指令存储/总线适配 │◄────►│                    │      │                │
└────────────────┘      │                    │◄────►│ 未来协处理器     │
                        │     rv32_core      │      │                │
┌────────────────┐      │                    │      └────────────────┘
│ 数据存储/总线适配 │◄────►│                    │
└────────────────┘      └────────────────────┘
                                  │
                                  ▼
                           退休跟踪与验证环境
```

## 2. 第一项系统级决定：指令与数据接口分离

处理器核使用相互独立的指令访问通道和数据访问通道。

这样做的原因是：

- IF 可以取指的同时，MEM 可以执行 load 或 store；
- 第一版不因单端口存储器产生额外结构冒险；
- 指令侧和数据侧可以分别连接简单存储器模型；
- 后续可以在核心外部分别连接 SRAM、Cache 或总线适配器；
- 五级流水本身不需要理解 AXI 的通道、突发和事务 ID。

“接口分离”描述的是核心的逻辑访问通道，不要求芯片最终必须具有两块完全独立的物理存储器。未来可以由核心外部的存储系统对两个通道进行仲裁。

## 3. 核心与外部模块的职责边界

### 3.1 `rv32_core` 负责

- 生成取指地址和数据访存地址；
- 发起指令读取、数据读取和数据写入；
- 在外部响应未完成时正确暂停相关流水级；
- 保证被冲刷或无效的指令不发出数据写请求；
- 保证同一条 store 或协处理器指令不会因 stall 重复发出副作用请求；
- 按程序顺序产生退休跟踪信息；
- 维护通用寄存器以及后续加入的 CSR 等架构状态。

### 3.2 核心外部的存储/总线适配器负责

- 把核心的逻辑访问请求转换为 SRAM、Cache 或 AXI 事务；
- 处理具体总线协议、仲裁、突发、缓冲和返回延迟；
- 向核心明确表示请求是否被接受、响应是否返回以及是否出现访问错误；
- 不依赖处理器内部某一级流水寄存器的实现细节。

### 3.3 验证环境负责

- 提供指令和数据存储器模型；
- 加载手写汇编、裸机程序和官方架构测试程序；
- 检查退休顺序、寄存器写回和数据存储器副作用；
- 在后续阶段注入等待周期和访问错误。

## 4. 顶层接口类别

`rv32_core` 顶层计划包含以下接口类别：

| 接口类别 | 方向 | 用途 |
|---|---|---|
| 时钟与复位 | 输入 | 驱动核心时序状态，并把核心带到合法初始状态 |
| 指令请求/响应 | 双向 | IF 发起取指，外部返回指令或取指错误 |
| 数据请求/响应 | 双向 | MEM 发起 load/store，外部返回 load 数据或访问错误 |
| 退休跟踪 | 输出 | 向验证环境报告真正退休的指令及其架构结果 |
| 协处理器命令/响应 | 双向、后续启用 | 发送自定义指令命令并接收执行结果 |

调试接口、JTAG、外部中断和 CSR 相关端口不进入当前这一轮定义。

## 5. 请求—响应事务模型

指令通道和数据通道都采用相互分离的请求、响应握手。请求方和响应方都使用 `valid/ready` 表示一次传输是否在当前时钟沿完成。

### 5.1 请求握手

请求只在时钟沿同时满足以下条件时被外部接受：

```text
request_fire = req_valid && req_ready
```

- 核心在具有一个待发送请求时置 `req_valid = 1`。
- 如果 `req_ready = 0`，核心必须保持 `req_valid` 和全部请求字段不变。
- 请求握手后，核心不得继续把同一请求作为新请求重复发送。
- 仅仅看见 `req_valid = 1` 不代表请求已经被外部接受。

### 5.2 响应握手

响应只在时钟沿同时满足以下条件时被核心消费：

```text
response_fire = rsp_valid && rsp_ready
```

- 外部在响应准备完成后置 `rsp_valid = 1`。
- 如果 `rsp_ready = 0`，外部必须保持 `rsp_valid` 和全部响应字段不变。
- 核心只有在响应握手后才能认为该事务已经完成。

### 5.3 在途事务限制

- 指令通道最多允许一笔已经被接受但尚未返回的请求。
- 数据通道最多允许一笔已经被接受但尚未返回的请求。
- 两个通道相互独立，因此可以各自同时存在一笔在途请求。
- 当前不使用事务 ID，所有响应都对应各自通道唯一的在途请求。
- 当前不允许数据请求乱序完成。

### 5.4 指令通道信息

指令请求至少包含：

```text
imem_req_valid
imem_req_ready
imem_req_addr
```

指令响应至少包含：

```text
imem_rsp_valid
imem_rsp_ready
imem_rsp_data
imem_rsp_error
```

核心需要在请求握手时保存该请求对应的 PC，使响应返回时能够把指令与正确的 PC 配对。

### 5.5 数据通道信息

数据请求至少包含：

```text
dmem_req_valid
dmem_req_ready
dmem_req_write
dmem_req_addr
dmem_req_wdata
dmem_req_wstrb
```

数据响应至少包含：

```text
dmem_rsp_valid
dmem_rsp_ready
dmem_rsp_rdata
dmem_rsp_error
```

`dmem_req_wstrb[3:0]` 分别控制 32 位写数据中的四个字节通道。load 和 store 都需要返回响应；store 的响应不携带有效读数据，但表示该事务已经结束。

核心必须区分“请求尚未被接受”和“请求已经被接受、正在等待响应”两种状态。特别是 store 请求握手后，即使流水线仍在等待响应，也不得再次发送同一个 store。

## 6. v0.1 存储器时序约束

v0.1 的验证存储器模型遵守以下约束：

```text
req_ready = 1
请求在周期 N 被接受
响应在周期 N+1 有效
rsp_error = 0
```

核心顶层保留完整的握手接口，但 v0.1 不把任意请求 backpressure、任意响应延迟和访问错误作为验收内容。v0.2 再放开这些约束并验证流水暂停、错误处理和精确异常。

## 7. 取指连续性与重定向

### 7.1 连续取指

v0.1 不增加独立的取指队列，IF/ID 是取指响应进入流水线的第一个存储位置。允许在同一个周期消费上一条指令响应并发送下一条顺序取指请求：

```text
C1：发送 PC0 请求
C2：接收 PC0 指令，同时发送 PC4 请求
C3：接收 PC4 指令，同时发送 PC8 请求
```

因此，在外部始终接受请求并固定一周期返回响应时，即使每个通道最多只有一笔在途请求，取指仍可达到稳定状态下每周期一条指令。

核心需要保存唯一在途取指请求对应的 PC。响应被正常接收时，将保存的 PC、响应指令和 `PC + 4` 一起写入 IF/ID。

### 7.2 前端暂停

当 ID stall 或其他原因导致 IF/ID 不能接收新指令时：

- IF/ID 保持不变；
- `imem_rsp_ready = 0`，不消费已经返回的响应；
- 外部保持 `imem_rsp_valid` 和响应字段不变；
- 核心不发送新的顺序取指请求。

暂停解除后，核心先消费被保持的响应，再恢复连续取指。

### 7.3 分支重定向与过期响应

当 EX 阶段产生 branch taken 时：

- 保存新的 `branch_target`，作为下一条有效取指地址；
- 清除 IF/ID 和 ID/EX 中的错误路径指令；
- 尚未握手的请求必须继续遵守 valid/ready 字段保持规则，不能在 `req_ready = 0` 时直接修改地址；
- 已被外部接受的错误路径请求不能取消，需要被标记为过期；
- 过期响应返回后必须完成响应握手，但不得写入 IF/ID；
- 过期事务排空后，从保存的 `branch_target` 重新开始取指；
- 如果过期响应恰好在 branch taken 当拍返回，可以当拍丢弃，并在满足单笔在途约束的前提下同时发送目标地址请求。

这种设计优先保证协议和功能正确性。可变长取指延迟可能增加分支代价，后续只有在性能分析证明必要时才增加取指队列、事务标签或多笔在途请求。

## 8. 数据访存期间的流水控制

### 8.1 EX 请求与 MEM 响应原则

EX 计算 load/store 地址和 store 写数据。数据请求在该指令从 EX 进入 MEM 的时钟边界完成握手，使固定一周期响应能够在该指令的 MEM 周期返回。

如果 `dmem_req_ready = 0`，load/store 不能进入 MEM，而是保持在 EX；PC、IF/ID 和 ID/EX 同时保持。更老的 MEM/WB 指令继续完成并排空。

请求握手后，load/store 进入 MEM，并记录请求已经发送。响应没有完成时，该指令保持在 MEM，所有更年轻的流水级停止向前推进。处理器不允许年轻指令绕过等待中的访存指令。

访存等待期间：

```text
PC       保持
IF/ID    保持
ID/EX    保持
EX/MEM   保持当前访存指令
MEM/WB   允许已有的更老指令完成一次，随后 valid 清零
```

不能把一个包含有效旧指令的 MEM/WB 永久保持，否则可能重复产生寄存器写回和退休事件。WB 中更老的指令正常完成一次后，等待期间 WB 保持为空。

### 8.2 数据事务状态

访存指令需要经历以下状态：

```text
EX：等待请求被接受
    ↓ request_fire
MEM：请求已经发送，等待响应
    ↓ response_fire
访存完成，可以进入 MEM/WB
```

- 请求尚未被接受时，保持 ID/EX、`dmem_req_valid` 和全部请求字段不变。
- 请求握手后记录“请求已经发送”，停止重复发送该请求。
- 等待响应期间，EX/MEM 中的地址、写数据和控制信息保持不变。
- 响应握手后清除事务等待状态，并允许该访存指令进入 MEM/WB。

上述状态可以在后续模块设计阶段实现为明确的 LSU 状态，也可以实现为功能等价的状态位；当前只冻结行为，不冻结具体 RTL 写法。

### 8.3 load 完成

- `dmem_req_addr` 来自 EX 的地址计算结果。
- load 响应握手后，把 `dmem_rsp_rdata` 与该指令的控制信息写入 MEM/WB。
- MEM/WB 中的 load 保持有效，下一周期在 WB 把按访存类型处理后的最终结果写回；字节/半字扩展具体放在 MEM 还是 WB，将在数据通路设计阶段确定。
- 在响应握手前，load 不得提前写回，也不得离开 MEM。

### 8.4 store 完成

- `dmem_req_addr` 来自 EX 的地址计算结果，`dmem_req_wdata` 来自经过前递选择的 `rs2_exec`。
- store 的数据写请求只允许握手一次。
- 请求握手后，即使流水线继续等待响应，也不得重新置位同一 store 的新请求。
- store 响应握手后，作为一条不写通用寄存器的有效指令进入 MEM/WB，用于保持按序退休和生成一次退休事件。
- 被 flush、无效或尚未到达 MEM 的 store 不得发出数据写请求。

### 8.5 与年轻分支的关系

如果更老的访存指令正在 MEM 等待，而更年轻的 branch 位于 EX：

- branch 保持在 EX，不允许重复产生 PC 重定向；
- branch taken 只有在 EX 指令获准向前推进时才成为一次有效控制事件；
- 如果数据响应正常返回与 branch taken 同周期发生，可以同时完成访存推进和分支重定向；
- 后续加入访问异常后，更老的 MEM 异常优先于更年轻的 branch taken。

### 8.6 一拍 load-use 时序

在 v0.1 固定一周期响应且 `dmem_req_ready = 1` 时：

```text
C1：load 在 EX 计算地址并完成请求握手，依赖指令在 ID 检出 hazard
C2：load 在 MEM 接收响应，EX 为 bubble，依赖指令继续留在 ID
C3：load 在 WB，依赖指令在 EX 通过 MEM/WB 前递获得 load 数据
```

因此只需插入一个 load-use bubble。如果未来在 EX→MEM 边界加入请求寄存器或请求缓冲，必须重新评估 load 延迟和冒险策略。

## 9. 时钟与复位边界

### 9.1 复位层次和极性

处理器系统采用以下复位分层：

```text
芯片级 ext_rst_n，低有效异步复位
              │
              ▼
      reset synchronizer
              │
              ▼
核心内部 rst，高有效同步复位
```

- 芯片或 FPGA 顶层输入使用低有效异步复位 `ext_rst_n`。
- reset synchronizer 负责异步进入复位、同步退出复位。
- `rv32_core` 只接收高有效同步复位 `rst`。
- 核心内部时序逻辑不直接把 `ext_rst_n` 放入异步敏感列表。
- 名称带 `_n` 的复位信号必须低有效；名称为 `rst` 的核心复位必须高有效。

### 9.2 核心复位状态

同步复位有效时，核心必须：

- 把 PC 设置为参数化的 `RESET_VECTOR`；
- 清除 IF/ID、ID/EX、EX/MEM、MEM/WB 的 `valid`；
- 清除取指请求在途、过期响应和重定向等待状态；
- 清除 LSU 请求已发送、等待响应等控制状态；
- 清除所有可能导致寄存器写、数据写、协处理器请求和退休事件的控制状态；
- 把 `imem_req_valid`、`dmem_req_valid` 和退休有效信号保持为 0；
- 不接受存储器响应，即复位期间 `imem_rsp_ready = 0`、`dmem_rsp_ready = 0`。

流水寄存器中的指令、PC 副本、操作数、ALU 结果和 store 数据等宽数据字段不要求复位。通用寄存器 `x1` 到 `x31` 不要求复位，`x0` 始终读出 0。

### 9.3 系统级复位约束

核心复位时，连接核心的指令/数据存储适配器必须同时清除旧事务，或者保证复位解除后不会返回复位前的响应。核心不能把复位前的响应与复位后的新请求配对。

`ext_rst_n` 释放后，reset synchronizer 等待规定数量的有效时钟沿再撤销核心 `rst`。核心可以在第一个采样到 `rst = 0` 的有效时钟沿接受 `RESET_VECTOR` 对应的首笔取指请求。

## 10. 退休跟踪接口

### 10.1 退休定义

一条指令完成全部架构行为，并且此后不可能再被暂停、冲刷或重新执行时，才算正式退休。当前五级流水统一在 WB 产生退休事件。

- ALU 指令在结果写回时退休。
- load 在数据响应完成并到达 WB 后退休。
- store 在完成响应后作为不写通用寄存器的有效指令到达 WB 并退休。
- branch 到达 WB 时退休，即使它不写通用寄存器。
- JAL/JALR 在链接地址写回时退休。
- bubble、无效指令、被 flush 的错误路径指令和复位期间不得产生退休事件。

### 10.2 v0.1 最小退休接口

```text
retire_valid
retire_pc
retire_instr
retire_rd_we
retire_rd_addr
retire_rd_data
```

- `retire_valid` 表示当前周期存在一条正常退休的指令。
- `retire_pc` 和 `retire_instr` 标识退休指令。
- `retire_rd_we` 表示该指令是否真正写通用寄存器。
- 只有 `retire_valid && retire_rd_we` 时，`retire_rd_addr` 和 `retire_rd_data` 才表示有效写回。
- 单发射核心每周期最多产生一条退休事件。
- 同一条指令只能产生一次退休事件。

### 10.3 访存副作用的验证

v0.1 不把地址、写数据和字节写使能额外携带到 WB。验证环境直接监视数据请求/响应通道，并把事务与按序退休事件关联：

```text
一次 store 请求握手
→ 一次 store 完成响应
→ 一次对应的退休事件
```

这样可以检查 store 是否丢失、重复发送或在错误路径上产生副作用，同时避免只为调试增加过多流水寄存器位宽。

### 10.4 v0.2 异常跟踪

v0.2 增加与普通退休语义分离的陷阱跟踪信息：

```text
trap_valid
trap_pc
trap_cause
trap_value
```

- `trap_valid` 表示 MEM 中的最老同步异常在当前周期正式提交。
- `trap_pc` 标识发生异常的指令，`trap_cause` 和 `trap_value` 分别对应将写入 `mcause` 和 `mtval` 的信息。
- 发生同步异常的指令只产生一次 trap 事件，并且永远不产生自己的普通退休事件。
- “retire 与 trap 互斥”只约束同一条指令，不约束整个周期。MEM 中的异常指令提交时，WB 中更老的正常指令可以在同周期产生一次 `retire_valid`。
- 验证环境在同周期同时观察到两类事件时，按程序年龄先记录 WB retire，再记录 MEM trap。
- 单发射核心每周期最多产生一条普通退休事件和一条 trap 事件；每条动态指令只能归属于其中一种结果。

该接口只报告已经提交的架构事件，不报告 IF、ID 或 EX 中尚可能被更老指令冲刷的原始异常。

## 11. 协处理器扩展边界

### 11.1 指令路径

- ID 识别 RISC-V custom 指令并读取 `rs1`、`rs2`。
- 协处理器指令随正常流水进入 EX。
- EX 在该指令有效、没有被 flush、没有更老的阻塞或异常，并且请求尚未发送时发起协处理器请求。
- 响应成功后，协处理器结果进入正常的 EX/MEM、MEM/WB、WB 路径。
- 核心使用正常写回端口更新 `rd`，不增加第二套通用寄存器写端口。

### 11.2 请求和响应信息

预留的请求信息为：

```text
cp_req_valid
cp_req_ready
cp_req_pc
cp_req_instr
cp_req_rs1_data
cp_req_rs2_data
```

预留的响应信息为：

```text
cp_rsp_valid
cp_rsp_ready
cp_rsp_data
cp_rsp_error
```

核心自己保存目的寄存器编号和写回控制，协处理器不返回 `rd`。

### 11.3 阻塞与单笔在途

- 每次最多允许一条协处理器指令在途。
- 请求握手后记录“请求已经发送”，等待期间不得重复请求。
- 响应返回前，协处理器指令保持在 EX，更年轻的 PC、IF/ID 和 ID/EX 保持。
- 更老的 MEM/WB 指令允许正常完成并排空。
- 协处理器响应完成后，该指令继续进入 EX/MEM。
- 如果更老的 MEM 指令正在等待或发生异常，协处理器指令不得提前发送请求。

### 11.4 未启用和错误路径行为

- v0.1 不实现协处理器 RTL，也不执行 custom 指令。
- 未启用协处理器时，custom 指令在异常功能完成后按非法指令处理。
- 被 flush、无效或无法向前推进的协处理器指令不得完成请求握手。
- `cp_rsp_error` 的异常语义在精确异常阶段定义。

### 11.5 NPU 命令与数据通路分离

协处理器接口只传递配置、命令、少量标量操作数和状态结果。NPU 的张量、权重和大块数据由独立 SRAM、AXI master 或 DMA 通路搬运，不通过 `rs1/rs2` 传输。

阻塞式命令接口不要求 CPU 等待完整 NPU 计算。未来的启动命令可以在 NPU 接受任务后快速响应，NPU 在后台运行，再通过状态轮询或中断报告完成。

当前只冻结此扩展边界，不创建协处理器 RTL，也不让协处理器功能进入 v0.1 的完成标准。

## 12. 本层结论

处理器核与外部世界的边界已经冻结：

- 单时钟和核心同步高有效复位；
- 独立的指令和数据请求/响应通道；
- 每个通道单笔在途，使用 valid/ready 握手；
- v0.1 固定一周期存储器响应；
- MEM 级数据访存反压和 store 防重复；
- 最小退休跟踪接口；
- 阻塞、单笔在途的协处理器扩展位置；
- SRAM、Cache、AXI 和 NPU 大数据通路位于核心外部。
