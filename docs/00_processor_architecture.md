# 处理器总体架构规格

> 文档状态：v0.1 草案  
> 当前阶段：项目目标与首版边界  
> 原则：先冻结整体架构，再进入模块接口和 RTL 实现。

## 1. 项目定位

本项目设计一颗可综合、可验证、可完成 ASIC 前端与综合后时序分析的 32 位 RISC-V 处理器核。处理器采用单发射、顺序执行、五级流水微架构，当前主线是掌握同步处理器的数据通路、流水控制、验证方法以及综合和 STA 闭环。

项目首先是一项学习型工程，其次才是功能实现。每项架构能力都应具备以下证据：

- 可以解释其架构语义和逐周期行为；
- 具有定向测试、关键波形和必要的断言；
- RTL 可综合，不依赖仅适用于仿真的写法；
- 能说明其面积、时序或功耗影响；
- 设计决定和已知限制记录在中文文档中。

## 2. 总体设计原则

- 正确性和可验证性优先于功能数量和频率指标。
- 采用自上而下设计：总体边界、微架构、模块接口、RTL、验证逐层推进。
- 核心数据通路保持工艺无关，SMIC 28nm 工艺库只用于综合和时序分析。
- 指令、数据、控制和副作用必须具有明确的流水级归属。
- 每一级流水状态都使用 `valid` 表示是否包含真实指令。
- 错误路径或无效指令不得产生寄存器写、存储器写和协处理器请求。
- 外部总线协议不直接侵入五级流水，后续通过适配器连接 SRAM、Cache 或 AXI。
- 为未来协处理器扩展保留架构位置，但当前不实现 NPU 数据通路。

## 3. 首个可运行版本 v0.1

### 3.1 微架构

- 32 位数据宽度和地址宽度；
- 单发射、顺序执行；
- `IF / ID / EX / MEM / WB` 五级流水；
- 指令和数据通路分离，避免单端口存储器造成结构冒险；
- 分支在 EX 阶段判断，采用静态不跳转策略；
- 支持 EX/MEM 和 MEM/WB 到 EX 的数据前递；
- load-use hazard 暂停一个周期；
- 单时钟域，核心内部采用同步复位；
- 复位入口地址使用参数 `RESET_VECTOR` 配置。

### 3.2 指令范围

v0.1 首先实现 RV32I 中构成基本整数程序所需的指令：

- 整数寄存器运算：`ADD、SUB、SLL、SLT、SLTU、XOR、SRL、SRA、OR、AND`；
- 整数立即数运算：`ADDI、SLTI、SLTIU、XORI、ORI、ANDI、SLLI、SRLI、SRAI`；
- 高位立即数：`LUI、AUIPC`；
- 条件分支：`BEQ、BNE、BLT、BGE、BLTU、BGEU`；
- 无条件跳转：`JAL、JALR`；
- 访存：`LB、LH、LW、LBU、LHU、SB、SH、SW`。

v0.1 的测试程序只使用自然对齐的取指和数据访问。非对齐访问、非法指令、`ECALL` 和 `EBREAK` 的精确异常行为在异常单元阶段补充。在此之前，不把 v0.1 宣称为完整的特权架构兼容实现。

### 3.3 首版运行环境

- 使用独立的指令存储器模型和数据存储器模型；
- 首次流水集成采用无等待或固定延迟的存储器响应；
- 通过手写汇编和小型裸机程序进行验证；
- 使用退休跟踪信息观察每条有效指令的 PC、指令字和写回结果。

### 4.1 v0.2 精确同步异常边界

v0.2 先建立可验证的同步异常闭环，再叠加中断。该阶段冻结以下边界：

- `FENCE` 是合法 RV32I 指令；在当前顺序、阻塞访存且无 Cache/store buffer 的微架构中作为保守的 no-op 进入流水并正常退休；
- 核心仍采用 RV32I 的 `IALIGN=32`，取指和控制转移目标必须按 4 字节对齐；
- 非对齐 load/store 产生精确异常，不向数据存储器发出请求；
- 非法指令、`ECALL`、`EBREAK`、取指/数据访问错误和地址未对齐都携带异常元数据随指令前进；
- 所有同步异常在 MEM 统一提交，使同一条指令只能“正常退休”或“产生 trap”二选一；
- trap 发生时允许同周期位于 WB 的更老指令先正常退休，再提交 MEM 中较年轻指令的 trap；
- `mepc`、`mcause`、`mtval` 和 trap 重定向构成最小 Machine Mode 异常状态闭环；
- 外部中断、优先级仲裁和 `mret` 的完整行为留到后续独立增量，不混入本轮同步异常实现。

本阶段的目标是先把异常的检测、传播、禁止副作用、统一提交和验证证据做完整，不同时扩大到 RV32M、外部总线或 NPU。

`FENCE.I` 属于独立的 Zifencei 扩展，不包含在本阶段。后续若增加 Cache、store buffer、多笔在途事务或允许数据通路修改指令存储器，必须重新评估 `FENCE`/`FENCE.I` 的实现，不能继续无条件沿用 no-op。

### 4.2 Zicsr 增量边界

Zicsr 增加 `CSRRW/CSRRS/CSRRC/CSRRWI/CSRRSI/CSRRCI` 六条原子 CSR 读改写指令。该扩展定义访问机制，但不单独规定必须存在的具体 CSR；本项目可访问的 Machine CSR 地址、字段和合法化规则由最小 Machine Mode 增量冻结。

实现按以下边界推进：

- 先独立验证三类 CSR 运算、寄存器/立即数源和读写抑制规则；
- CSR 指令在 ID 译码，在 EX 固化源操作数，在 MEM 对当前 CSR 值执行原子读改写并提交；
- 普通 CSR 写与同步异常自动更新共用唯一状态所有者；异常指令禁止显式 CSR 写，由该所有者完成 trap 自动更新，异步中断与同周期更新顺序留到 Machine interrupt 增量冻结；
- CSR 返回的旧值经 MEM/WB 写入 `rd`，不增加通用寄存器堆写端口；
- 不存在、权限不足或对只读 CSR 发起真实写操作时，产生 illegal-instruction 异常且不产生 CSR/通用寄存器副作用；
- 本增量不声明 Zicntr、Zihpm 或任何尚未实现的 CSR 集合。

本项目 v0.2 实际存在的 CSR 地址、reset、WARL/MRO 规则、trap 自动更新和分阶段
延期范围，以 [Machine CSR Profile 与状态所有者契约](06_machine_csr_contract.md) 为准。
状态所有者、最终异常合并和六条指令的端到端验证已经完成；主 decoder 只对
`rv32_csr_decoder` 明确认可的六种 Zicsr 编码清除 illegal，具体 CSR 地址合法性仍在
MEM 由唯一状态所有者判定。

## 5. 当前明确不做的内容

- 乱序执行、超标量和多发射；
- 多核和缓存一致性；
- MMU、虚拟内存和 Linux 启动；
- 动态分支预测；
- 一级 Cache；
- 非阻塞访存和多个在途访存请求；
- 浮点和向量扩展；
- 当前阶段的 NPU RTL；
- 在核心内部直接实现 AXI 协议；
- 在功能稳定前追求极限频率或过早进行局部 PPA 优化。

上述内容是当前项目的范围控制，不代表架构永久禁止扩展。

## 6. 首版完成标准

v0.1 只有同时满足以下条件才算完成：

- 规定的指令均有定向测试；
- RISC-V 官方架构测试集中与当前已实现 ISA 子集匹配的测试全部通过；
- 官方测试所使用的测试集版本、配置、工具链版本和结果必须记录，不能通过临时忽略失败用例来宣称通过；
- 数据冒险、load-use、分支冲刷和 store 副作用均有关键波形；
- 错误路径指令不会写寄存器或数据存储器；
- 退休顺序与参考执行结果一致；
- RTL 不存在锁存器、多驱动和不可综合语句；
- VCS 回归稳定通过；
- SpyGlass 不存在未解释的高严重级问题；
- DC 可以完成综合，且不存在未约束的关键时序路径；
- 关键设计决定、限制和验证结果已记录。

官方架构测试主要验证指令的架构语义，不能替代处理器内部的定向测试、流水冒险测试、随机延迟测试、断言和波形检查。
