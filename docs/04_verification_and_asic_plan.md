# 验证体系与 ASIC 阶段验收计划

> 上位规格：`00_processor_architecture.md`、`01_core_system_context.md`、`02_pipeline_contract.md`、`03_module_architecture.md`  
> 当前目标：为 v0.1 五级流水核心建立可重复、可解释、可逐层收敛的验证与 ASIC 验收体系。  
> 协作方式：用户手写 RTL 和主要 testbench；助手负责讲解原理、拆分任务、设计检查点、审阅代码并协助定位问题。

## 1. 验证目标

本项目不以“某个程序跑通”作为处理器正确性的充分证据。每项功能至少从以下四个角度验证：

1. **局部逻辑正确**：叶子模块对输入产生正确输出；
2. **逐周期行为正确**：流水推进、暂停、冲刷和握手符合已冻结契约；
3. **架构结果正确**：退休顺序、寄存器结果和存储器副作用符合 RV32I；
4. **ASIC 实现合法**：RTL 可 lint、可综合、综合前后等价且时序约束完整。

验证环境首先服务于学习和定位错误，因此首版采用直接、透明的 SystemVerilog testbench，不引入 UVM、class 或复杂验证框架。只有当测试规模确实需要复用和随机化时，才重新评估验证方法学。

## 2. 验证分层

### 2.1 第 0 层：编译与静态检查

每次提交首先检查：

- `rv32_pkg.sv` 在所有引用它的 RTL 之前编译；
- RTL 和 testbench 均能通过 VCS 编译；
- 不存在重复定义、多驱动、隐式网络和明显位宽错误；
- `always_comb` 不推断锁存器；
- `always_ff` 中的状态只有唯一时序驱动；
- 仿真启动后不存在无法解释的 `X/Z` 扩散。

该层失败时不进入更高层回归。

### 2.2 第 1 层：叶子模块单元测试

优先独立验证纯组合模块和局部状态模块：

| 模块 | 主要检查内容 |
|---|---|
| `rv32_imm_gen` | I/S/B/U/J 五类立即数的拼接和符号扩展 |
| `rv32_decoder` | 指令到语义控制的映射、源寄存器使用标记、非法编码 |
| `rv32_alu` | 算术、逻辑、移位以及 signed/unsigned 行为 |
| `rv32_branch_compare` | 六类条件分支的有符号和无符号比较 |
| `rv32_regfile` | 双读单写、`x0` 恒零和同步写入；WB→ID 同周期旁路在 `rv32_idu` 中验证 |
| `rv32_forward_unit` | 两级前递优先级、`x0` 排除、load-use 检测 |
| `rv32_pipeline_ctrl` | 每种事件对应的 `LOAD/HOLD/CLEAR` 动作和优先级 |
| `rv32_lsu` 局部功能 | load 扩展、store 数据移位、`wstrb` 和地址低位处理 |

单元测试尽量采用“输入、期望值、立即比较”的形式。对枚举和边界值应显式列举，不只依赖少量随机输入。

### 2.3 第 2 层：核心定向测试

核心级 testbench 连接独立的指令存储器模型和数据存储器模型，通过手写汇编程序验证完整流水行为。

定向测试分为四组：

1. **单指令语义**：每条已实现 RV32I 指令至少一个正常用例和必要边界用例；
2. **数据冒险**：EX/MEM 前递、MEM/WB 前递、WB→ID 旁路、连续生产者优先级和 load-use；
3. **控制冒险**：taken/not-taken branch、JAL、JALR、重定向目标和错误路径清除；
4. **访存副作用**：不同宽度的 load/store、store data 前递以及一次且仅一次的写入。

每个测试应具有明确结束条件和超时条件，不能依赖人工观察“仿真似乎停止”。测试结束时至少检查：

- 期望的寄存器或签名区结果；
- 退休指令序列；
- 数据存储器最终内容；
- 断言没有失败；
- 没有超时和无法解释的未知值。

### 2.4 第 3 层：协议与停顿测试

v0.1 的正式功能验收使用 `req_ready = 1`、请求后一周期响应的固定时序，但 RTL 顶层已经保留完整 valid/ready 协议。因此还需要少量协议定向测试，证明接口没有明显违反契约：

- 请求未被接受时，`req_valid` 和请求字段保持不变；
- 响应未被接收时，响应方保持 `rsp_valid` 和响应字段不变；
- 指令前端暂停时不错误消费响应；
- 已接受的错误路径取指响应被排空但不进入 IF/ID；
- store 请求只握手一次，等待响应时不重复发送；
- MEM 等待时年轻流水级保持，较老 WB 指令只退休一次；
- reset 清除在途事务状态，复位前响应不能污染复位后执行。

任意长等待和广泛随机 backpressure 属于 v0.2 验收内容。v0.1 只要求用有限的定向等待验证接口基本健壮性，不因此扩大首版范围。

### 2.5 第 4 层：RISC-V 官方架构测试

运行与 v0.1 已实现 RV32I 子集相匹配的官方架构测试。执行时必须记录：

- 测试仓库及精确版本；
- RISC-V 工具链及精确版本；
- 编译选项、ISA 配置和链接地址；
- 被选中的测试列表；
- 每个测试的 PASS/FAIL 结果；
- 被排除测试的明确架构原因。

不允许为了得到全绿结果而临时跳过本应支持的失败测试。官方测试用于验证 ISA 可见结果，但不替代数据冒险、冲刷、握手和 store 副作用的定向测试。

### 2.6 第 5 层：退休级参考模型比对

核心通过以下信息报告每条普通退休指令：

```text
retire_valid
retire_pc
retire_instr
retire_rd_we
retire_rd_addr
retire_rd_data
```

验证环境按照程序顺序记录退休流，并与锁定版本的参考模型结果比较。首版可以先离线比较日志，后续再升级为仿真过程中的在线 scoreboard。

仅比较寄存器写回不足以覆盖 store，因此数据通道还要独立记录成功握手的写请求及其最终副作用。参考模型、RTL 和测试环境必须使用相同的初始存储器映像。

## 3. v0.1 必须具备的核心定向用例

### 3.1 基本指令

- 所有 R 型整数指令；
- 所有已支持 I 型整数指令；
- `LUI`、`AUIPC`；
- 六类条件分支的 taken 和 not-taken；
- `JAL`、`JALR`，包括 JALR 目标最低位清零；
- `LB/LBU/LH/LHU/LW`；
- `SB/SH/SW`。

需要特别覆盖 `0`、`1`、`-1`、最高位为 1、最大正数、最小负数以及移位量边界。

### 3.2 数据相关

至少包含以下指令距离：

```text
生产者
消费者             EX/MEM 前递

生产者
独立指令
消费者             MEM/WB 前递
```

还必须覆盖：

- 同一消费者的 `rs1`、`rs2` 同时相关；
- 连续两条指令写同一 `rd` 时，消费者选择较新的 EX/MEM 结果；
- 生产者目的寄存器为 `x0` 时不得前递；
- ALU 结果前递给 branch 比较；
- ALU 或 load 结果前递给 store data；
- WB 写入与 ID 读取同一寄存器时，ID 得到 WB 新值；
- load-use 恰好插入一个 bubble，并从 MEM/WB 获得 load 数据。

### 3.3 控制相关

- not-taken branch 不冲刷顺序路径；
- taken branch 清除 IF/ID 和 ID/EX 中的年轻指令；
- 错误路径指令不得写寄存器、写存储器或退休；
- branch 的比较操作数来自前递时仍能正确重定向；
- JAL/JALR 把 `PC+4` 写入正确的 `rd`；
- JALR 同时验证源寄存器前递和目标最低位清零；
- 重定向与取指响应同周期出现时，错误路径响应被丢弃；
- 更老 MEM 等待时，年轻 branch 不得重复重定向。

### 3.4 访存相关

- byte/half/word 在各合法字节位置的读写；
- signed load 和 unsigned load 的扩展差异；
- store 写数据及 `wstrb` 与地址低位匹配；
- load 请求、响应、写回和退休只各发生一次；
- store 请求和外部写副作用只发生一次；
- store 在收到响应后才退休；
- 被冲刷的 store 不产生数据请求；
- 数据请求等待或响应等待期间，请求字段及流水状态符合契约。

## 4. 断言计划

断言分为“设计不变量”和“协议时序”。初期可放在 testbench checker 中；当层次稳定后，再决定哪些断言绑定到 RTL 层次。断言仅用于检查，不参与综合功能。

### 4.1 设计不变量

- `x0` 的架构读取值永远为零；
- `rd_addr == 0` 时不得产生真实寄存器写副作用；
- 无效流水项不得写寄存器、发出数据请求或退休；
- 每周期最多退休一条指令；
- `retire_rd_we` 有效时必须同时满足 `retire_valid`；
- 被冲刷指令永远不能在未来退休；
- load 在响应完成前不得写回；
- store 在响应完成前不得退休；
- 同一条 store 不得产生两次请求握手或两次存储器写副作用；
- 协处理器禁用时 `cp_req_valid` 和 `cp_rsp_ready` 恒为零。

### 4.2 valid/ready 协议

- `req_valid && !req_ready` 时，下周期请求 valid 和全部请求字段保持；
- `rsp_valid && !rsp_ready` 时，下周期响应 valid 和全部响应字段保持；
- 没有在途请求时不得消费对应响应；
- 单通道在途事务数只能为 0 或 1；
- 一次请求握手只能对应一次响应握手；
- reset 有效期间请求、响应接收和退休均无效。

### 4.3 流水控制

- `HOLD` 时对应流水寄存器的全部字段保持；
- `CLEAR` 后对应流水寄存器 `valid` 为零；
- load-use 事件使 PC、IF/ID 保持，并清除 ID/EX；
- qualified redirect 只在流水控制选中 EX redirect 时产生；
- MEM wait 的优先级高于年轻 redirect 和 load-use；
- MEM wait 期间 MEM/WB 中原有老指令最多完成一次，随后保持为空。

断言失败必须定位根因。不得用全局关闭断言或宽泛 waiver 把未理解的问题变成“通过”。

## 5. Testbench 基本结构

v0.1 建议使用以下轻量结构：

```text
tb_rv32_core
├── clock/reset driver
├── instruction memory model
├── data memory model
├── program/image loader
├── retire monitor
├── memory-side-effect monitor
├── assertions/checkers
└── test control and timeout
```

存储器模型的协议行为与存储内容分开实现，便于后续从固定一周期响应替换为可配置等待。testbench 不通过窥探内部寄存器来判断程序正确；架构结果优先通过退休接口和外部存储器副作用检查。内部层次信号只用于断言、调试和波形解释。

## 6. 回归分级与证据

### 6.1 Smoke 回归

用于日常快速检查，包含：

- 编译和基础 lint；
- 每个已完成叶子模块的少量代表用例；
- 一个基本算术程序；
- 一个数据冒险程序；
- 一个 taken branch 程序；
- 一个 load/store 程序。

目标是快速发现接口破坏和主要功能回退。

### 6.2 Full 回归

用于阶段验收，包含：

- 全部叶子模块测试；
- 全部核心定向测试；
- 协议和停顿测试；
- 当前适用的官方架构测试；
- 退休级参考结果比较；
- 全部断言。

### 6.3 每个里程碑保存的证据

- 可重复执行的命令或脚本；
- 工具版本、测试配置和随机种子；
- 汇总 PASS/FAIL 日志；
- 首个错误的详细日志；
- 关键波形截图或保存的波形数据库；
- 已知限制和有依据的 waiver；
- 对失败根因及修复方式的简短记录。

波形不是“信号很多”的截图，而应能回答一个明确问题，例如：为什么 load-use 只停一拍，或者为什么错误路径 store 没有发出请求。

## 7. ASIC 工具流程与质量门槛

主目标工艺采用 SMIC 28nm。SMIC 180nm 可以在核心稳定后作为工艺尺度对比练习，但不同时维护两套主收敛目标，以免分散精力。

本章描述项目最终要跑通的完整工具链，但不表示所有工具都属于 v0.1 完成条件：

- v0.1 完成 VCS 功能回归、SpyGlass 静态质量检查和 DC 基础综合；
- v1.0 在稳定的 v0.1 RTL 基础上，进一步完成可重复的 FM 等价验证、PT 布局前 STA 和代表性门级仿真闭环。

### 7.1 VCS：RTL 功能基线

进入综合前必须满足：

- smoke 和 full 回归稳定通过；
- 没有断言失败和仿真超时；
- 没有无法解释的 `X/Z`；
- 测试结果不依赖用例执行顺序；
- 固定随机种子可以复现问题。

Verdi 用于观察流水 valid、PC、指令、前递选择、流水动作、请求/响应握手和退休事件。每类关键冒险至少保留一份可讲解波形。

### 7.2 SpyGlass：RTL 静态质量

重点检查：

- latch、多驱动、组合环路；
- 未连接端口、未使用信号和不可达逻辑；
- 位宽截断、符号扩展和 signed/unsigned 混用；
- reset 使用和时钟使用；
- case 完整性和 enum 合法性；
- 可能导致仿真与综合不一致的写法。

高严重级问题必须清零或逐条给出设计依据。waiver 必须足够局部，写明对象、原因和为什么不会影响功能，不能使用宽泛规则屏蔽整个模块。

### 7.3 DC：综合与时序结构检查

综合前准备：

- 锁定 RTL 文件清单和顶层模块；
- 读取正确的 SMIC 28nm 标准单元库；
- 定义时钟、输入输出延迟和必要环境约束；
- 声明真正的 false path 或异步路径，不能为消除违例而随意切断路径；
- 保存工具版本、库角和综合脚本。

首次综合不预先承诺极限频率。先使用宽松且合理的时钟周期验证约束和结构正确，再依据关键路径报告逐步收紧。通过标准为：

- elaborate、link 和 compile 成功；
- 无未解析引用和意外 black box；
- 无锁存器和异常时钟门控；
- 所有关键时序路径均已约束；
- setup 约束在选定目标和库角下收敛；
- 面积、寄存器数量和组合单元构成可解释；
- 保存综合网表、DDC、SDC 和报告。

### 7.4 Formality：RTL 与综合网表等价

FM 比较 RTL 与 DC 综合网表。通过标准为：

- reference 与 implementation 正确读入并匹配；
- 时钟、复位、参数和常量设置与综合一致；
- 所有需要验证的 compare point 成功；
- 不存在未解释的 failing、aborted 或 unmatched point；
- 综合优化、未复位数据寄存器和常量传播造成的提示均能说明原因。

FM 证明的是给定约束下 RTL 与综合网表等价，不证明 RTL 本身符合 ISA，因此不能替代功能仿真。

### 7.5 PT：综合网表静态时序分析

PT 读取综合网表、库、SDC 和连线寄生信息。当前没有布局布线工具，因此项目先完成：

- **综合后、布局前 STA**：使用 DC 综合网表和线负载/估算互连；
- 检查 setup、hold、最大转换、最大电容和最大扇出；
- 检查时钟定义、端点覆盖和未约束路径；
- 分析关键路径经过的数据通路和控制逻辑；
- 验证例外路径的必要性和精确性。

在没有布局布线和提取寄生参数的情况下，不能把该结果称为 post-layout signoff。若后续获得 P&R 工具和 SPEF，再增加布局后 STA、串扰和多模式多角分析。

### 7.6 门级仿真

门级仿真不是每次 RTL 修改的日常回归。完成 DC 和 FM 后选择少量 smoke 程序进行：

- 零延迟门级仿真，检查综合网表连接和初始化假设；
- 条件具备时进行 SDF 回标仿真，检查 reset 和基本时序行为。

门级仿真不能替代 STA；FM 已完整通过时，也不需要用大量门级用例重复 RTL 全回归。

## 8. 阶段门禁

| 门禁 | 进入条件 | 通过证据 |
|---|---|---|
| G0：模块可集成 | 包和接口编译通过 | 编译日志、静态检查结果 |
| G1：局部逻辑正确 | 对应叶子 RTL 完成 | 单元测试 PASS、边界用例 |
| G2：核心基本运行 | 五级流水首次连通 | smoke 回归、退休日志、关键波形 |
| G3：v0.1 功能冻结 | 所有范围内功能实现 | full 回归、官方测试、断言、参考结果 |
| G4：RTL 质量完成 | G3 稳定 | SpyGlass 报告和受控 waiver |
| G5：v0.1 工程完成 | RTL 与基础约束冻结 | DC 网表、约束、面积与时序报告 |
| G6：v1.0 等价完成 | DC 输出稳定且流程可重复 | FM 全部 compare point 通过 |
| G7：v1.0 布局前 STA 完成 | 网表、库、约束一致 | PT coverage、setup/hold 与 DRC 报告 |
| G8：v1.0 闭环完成 | G6、G7 通过 | 代表性门级 smoke、归档脚本和完整报告 |

后一级门禁不能替代前一级。例如，FM 通过不能弥补官方测试失败，PT 时序通过也不能弥补未约束路径。

## 9. 里程碑完成判据

### 9.1 v0.1 完成判据

只有同时满足以下条件，才把五级流水核心 v0.1 标记为完成：

- 范围内每条指令具有定向测试；
- 所有规定的数据、控制和访存冒险测试通过；
- 关键设计不变量和 valid/ready 协议断言通过；
- 适用的 RISC-V 官方架构测试全部通过，版本和排除原因记录完整；
- 退休顺序、寄存器结果和 store 副作用与参考结果一致；
- VCS full 回归可重复通过；
- SpyGlass 没有未解释的高严重级问题；
- DC 综合成功且关键路径均已约束；
- 每个关键机制都有可讲解的波形和设计记录。

这里的 DC 目标是证明 RTL 可综合、基础约束有效、面积和关键路径可解释，不要求 v0.1 已经完成全部综合后验证闭环。

### 9.2 v1.0 完成判据

v1.0 在 v0.1 全部完成条件之上，还必须满足：

- VCS、SpyGlass、DC、FM 和 PT 脚本能够从已记录的版本和配置重复执行；
- FM 证明冻结 RTL 与 DC 综合网表等价；
- PT 完成布局前 STA，不存在未解释的未约束路径、时序违例或设计规则违例；
- 代表性零延迟门级 smoke 通过；条件具备时完成 SDF 回标 smoke；
- 网表、DDC、SDC、工具报告、waiver 和关键版本信息归档完整。

由于当前没有布局布线工具和提取寄生参数，v1.0 的 PT 结果仍是布局前 STA，不能称为 post-layout signoff。

## 10. 下一步实施顺序

架构层文档完成后，RTL 不从完整 `rv32_core` 一次写起，而按可验证依赖逐步建立：

1. 建立目录、文件清单和编译骨架；
2. 手写 `rv32_pkg.sv` 的常量、枚举和 packed struct；
3. 为公共包做编译检查和类型审阅；
4. 手写并单测无状态叶子模块；
5. 手写 regfile、IFU、LSU 等有状态模块并做局部验证；
6. 最后在 `rv32_core` 中连接流水寄存器和全局控制；
7. 核心功能稳定后先用 SpyGlass 和 DC 完成 v0.1，再用 FM、PT 和代表性门级仿真完成 v1.0 闭环。

下一轮从第 1、2 项开始，但每一段 RTL 仍由用户亲手输入。助手先解释该段代码表达的硬件、接口和常见错误，再由用户编写并提交审阅。
